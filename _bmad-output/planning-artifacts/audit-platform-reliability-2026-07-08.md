# Platform Reliability Audit (Non-AI Layer)

**Date:** 2026-07-08
**Scope:** Everything **except** the AI layer, which was audited separately in `audit-ai-layer-reliability-2026-07-08.md` (findings RC/S/C/H/M/X → epics R1–R11). This pass covers: core infra (`server.py`, `database.py`, `tenant.py`, auth middleware, `school_context.py`), the non-AI HTTP routes, the billing/tokens/Razorpay path, the auth & upload surfaces, and the shared services layer.
**Method:** Deep-read of core infra + auth + billing + upload + file-serve paths; targeted pattern sweeps (`get_raw_db`, unscoped Mongo methods, N+1, branch-scope comments) across all 28 route files; spot verification of the highest-severity claims.
**Companion docs:** `epics-platform-reliability.md` (epics R12–R15, continuing the AI-layer numbering), `architecture.md`.

> **Portability note:** Written model-agnostically like the AI-layer audit. Every finding carries exact `file:line` and a reproduction so any executing agent can act without re-deriving context. Severity rubric identical to the AI-layer audit (Critical → Low).

---

## 1. Headline: the platform's single tenant assumption is now a liability

The AI-layer audit's systemic conclusion was "no turn-completion contract." The platform's equivalent systemic finding is: **the codebase was built single-tenant (one school, `SCHOOL_ID` env var) and the multi-school SaaS onboarding path added in Story 7-44/7-45 (`operator.py create_school`) is wired on top without the tenant-context plumbing it needs.** The most severe findings below (P-C1, P-C2, P-H1) are all instances of this: a code path that is correct for The Aaryans today but silently wrong the moment a second school is provisioned. Because onboarding *exists in the product*, these are live latent defects, not hypotheticals.

## 2. Critical

- **P-C1 — A newly provisioned school owner cannot log in.** `backend/routes/operator.py:280-290` (`create_school`) inserts the owner `auth_users` row with `"username": owner_email` but **no `username_lower` field**. Login (`backend/routes/auth.py:184`) looks users up exclusively by `{"username_lower": username_lower}`, and the unique index is on `(username_lower, schoolId)` (`database.py:352-358`). Result: `find_one` returns nothing → `401 Invalid username or password` for every freshly onboarded owner, and the null `username_lower` collides on the unique index if a second school is created the same way. The entire SaaS onboarding flow is broken at the first login. (Also missing vs. the login/refresh path: `user_info` sub-document, `initials`, `branch_id` — `_jwt_payload_from_auth` reads `user_info` first, so even if login found the row the JWT would be thin.)

- **P-C2 — Razorpay webhooks credit the wrong tenant under multi-school.** All four webhook handlers (`services/razorpay_service.py:143-374`) call `get_db()`, which scopes writes by `get_school_id()`. The webhook endpoint (`routes/tokens.py:242`) is **unauthenticated by design** (signature-verified, no Bearer token), so `SchoolContextMiddleware` never sets the per-request school context → `get_school_id()` falls back to the **env-default `SCHOOL_ID`**. Every `token_purchases` / `token_balances` write therefore lands in the default tenant's scope regardless of which school's branch actually paid. For The Aaryans-only today this is invisible; the instant a second school pays, its top-ups/subscriptions credit the wrong tenant's collection. The `branch_id` inside `notes` is trusted but the `schoolId` scoping is not derived from it.

## 3. High

- **P-H1 — First-ever top-up for a branch raises a Mongo path conflict.** `services/razorpay_service.py:349-364` (`purchase_topup_razorpay`, and identically `handle_subscription_charged:278-287`) issues `{"$inc": {f"personal_topups.{user_id}": tokens}, "$setOnInsert": {..., "personal_topups": {}}}` with `upsert=True`. MongoDB rejects an update that both `$inc`s a sub-path of `personal_topups` **and** `$setOnInsert`s `personal_topups` itself — "Updating the path 'personal_topups' would create a conflict at 'personal_topups'." So the first paid top-up for a branch whose `token_balances` doc does not yet exist throws inside the webhook handler → caught by the `tokens.py:272` blanket `except` → **200 returned to Razorpay, no tokens credited, no retry**. Customer paid, got nothing.

- **P-H2 — Paid-but-not-credited window (non-atomic credit).** Both credit paths insert the `token_purchases` idempotency row (`insert_one`), then `update_one` the balance as a **separate, non-transactional** operation (`razorpay_service.py:333-364`, `262-287`). If the process dies between the two, the purchase is recorded (so the `DuplicateKeyError` idempotency guard now blocks any retry) but the balance was never incremented. Money in, tokens permanently lost, and the webhook redelivery is silently deduped. Credit must be atomic (single transaction) or the idempotency claim must be written *after* a successful, idempotent balance increment.

- **P-H3 — File-serve and file-list leak across admin sub-roles.** `routes/upload.py:208` (`serve_file`) and `:233` (`list_uploads`) gate on `user.get("role") not in ["owner", "admin"]`. Any `admin` sub_category — accountant, receptionist, maintenance, IT-tech — can therefore fetch a presigned URL for, and enumerate, **every uploaded file in the school**, including files uploaded by the owner, principal, or teachers (fee documents, student photos, staff docs). This contradicts the fine-grained `require_access(sub_category=...)` model used elsewhere and the DPDP least-exposure directive.

- **P-H4 — Bulk PII export open to every admin sub-role, and teachers can export the whole school's attendance.** `routes/exports.py`: `export_students`/`export_staff`/`export_enquiries` use `require_role("owner", "admin")` — no sub_category gate — so any admin sub-role can download the full student/staff/enquiry PII CSV. `export_attendance` (`:104`) allows `teacher` with **no branch/class scoping** (`scoped_filter` is school-wide), letting any teacher export all 10 000 attendance rows for every class in the school. `_require_owner_or_accountant` (`:16-23`) also still honors the legacy `"accounts"` sub_category that the AI-layer audit's C4 identified as non-canonical (`"accountant"`).

## 3b. High (surfaced by the R15 line-level read)

- **P-H6 — Two divergent payroll-disbursement implementations write the same collection.** `routes/payroll.py:101` (`POST /api/payroll/disburse`, owner-only) and `routes/fees.py:739` (`POST /api/payroll/disbursements`, owner + admin/accountant) both insert into `db.salary_disbursements` with **incompatible schemas** — payroll.py writes `gross`/`net`/`disbursed_by`; fees.py writes `base_salary`/`allowances`/`deductions`/`net_amount`/`paid_by` — and **incompatible idempotency**: payroll.py relies on the `(schoolId, staff_id, month)` unique index and catches `DuplicateKeyError` (→409), while fees.py does an unguarded find-then-insert (`fees.py:753`) that races and, on a concurrent double-submit, hits the same unique index **without catching it → 500** (and burns a random `_id`≠`id` pair, `fees.py:759-760`). Net effects: a disbursement created via one route is misread by the other (blank/₹0 salary where a consumer reads the missing field), two different auth policies gate the same money action, and the "idempotent" fee route can 500 instead of deduping. There are likewise two salary-*structure* upserts (`payroll.py:47` owner-only vs `fees.py:706` owner/admin) with different schemas. One canonical disbursement path + schema is needed (mirror the fee-payment single-service pattern in `fees_service.record_payment`).

- **P-H7 — Deactivating a staff member does not revoke their sessions.** `routes/staff.py:180` (`delete_staff`) runs `db.refresh_tokens.update_many({"user_id": ..., "revoked": False}, {"$set": {"revoked": True, ...}})` — but the refresh-token schema has **no `revoked` boolean**; active tokens are marked by `revoked_at: None` (`services/auth_tokens.py:115,167`). The filter `{"revoked": False}` therefore matches **zero documents**, so a deactivated staff member's refresh token stays valid and they can keep minting access tokens for up to the 7-day TTL. The correct helper `revoke_user_refresh_tokens(db, user_id)` already exists and is used by `operator.deactivate_school`; `delete_staff` should call it instead of hand-rolling a query against a field that doesn't exist.

## 4. Medium

- **P-H5 (deep-read follow-up) — Operations lists leak across branches.** `routes/operations.py` mixes `scoped_query(branch_id=...)` (correct: `list_incidents:501-502`) with plain school-wide `scoped_filter` on peer endpoints: `list_leave_requests` (`:219-220`), `list_approval_requests` (`:327`), and complaints (`:1240`) — and the expenses/visitors/assets/transport lists — apply no branch filter, so a branch-A admin sees branch-B leave requests, approvals, complaints, expenses, and assets. This is the "Wave-3 branch-isolation" work CLAUDE.md already flags as in-progress; this audit confirms the specific endpoints and rules it in-scope. Each hit must either migrate to `scoped_query(branch_id=user.get("branch_id"))` or carry a `# branch-scope: intentional — <reason>` comment per the CLAUDE.md audit rule.

- **P-M6 — Unescaped user input interpolated into `$regex` (injection / ReDoS / 500).** Several search/filter endpoints build regexes from raw request params without `re.escape`: `audit.py:78-81` (`q` → changed_by/entity_id/action), `operations.py:492-494` (`q` → incident description/parties), and format-string period filters `fees.py:1010` (`^{period}`), `academics.py:631,638` (`^{current_month}`), `attendance.py:347` (`^{month}`). A crafted value like `q=(a+)+$` triggers catastrophic backtracking (CPU DoS by an authenticated admin/owner), and a value like `q=[` yields a malformed-regex `500`. Contrast the already-correct `students.py:181` and `search.py:98` which `re.escape`. Fix: `re.escape` all user-supplied regex operands, and validate period/month/week params against a strict `YYYY-MM` (or `-W\d+`) pattern before interpolation.

- **P-M7 — Bulk SMS/WhatsApp send to client-supplied targets, unscoped, with no cost cap.** `routes/sms.py` `send_bulk` (`:123`) and `whatsapp-fee-reminders`/`whatsapp-attendance-alerts` (`:423,495`) send to phone numbers taken verbatim from the request body — nothing ties a recipient to an actual student/guardian of the school — so any admin can drive the school's Twilio account to send to arbitrary numbers (spam/cost abuse); the only bound is 500/request with no per-day cap. `send_parent_message` (`:216-222`) looks up `db.students.find_one({"id": sid})` and its guardian **without branch scoping**, so a branch-A admin can message branch-B parents by supplying their student IDs. `get_sms_logs` (`:290`) is school-wide, exposing every branch's parent phone numbers and message bodies to any branch admin. Add student/guardian ownership validation, branch scoping, and a per-school daily send cap.

- **P-M8 — Bulk-imported students are orphaned from branch scoping and written non-atomically.** `routes/import_data.py` `_student_doc` (`:164-182`) omits `branch_id`, so every student created by `commit_import` has no branch field and is therefore invisible to every `scoped_query(branch_id=...)` read and unmanageable by branch-bound admins. `commit_import` (`:243-279`) also inserts the student and its guardian as two separate un-transacted writes (partial failure → a student with no guardian) and is not idempotent across re-submitted files beyond the name+class duplicate heuristic. The class lookup (`:112`) and duplicate check (`:119`) are likewise branch-unscoped.

- **P-M1 — `ScopedCollection` silently does not scope five write/read methods.** `database.py:60-112` overrides `find`, `find_one`, `count_documents`, `insert_*`, `update_*`, `delete_*`, `aggregate` — but **not** `find_one_and_update`, `find_one_and_delete`, `find_one_and_replace`, `replace_one`, `bulk_write`, or `distinct`. These fall through `__getattr__` to the raw Motor collection with **no `schoolId` injection**. Today only `fees.py:884` (`find_one_and_update` on `receipt_counters`, manually keyed by `{school_id}-{period}` so no cross-tenant leak) and `ai_rate_limiter.py` (counters) use them, so impact is currently contained — but this is a **latent tenant-isolation trap**: the next developer who reaches for `find_one_and_update` gets an unscoped query with no warning. Add overrides, or make the un-overridden methods raise.

- **P-M2 — Login lockout and multi-school lookup are keyed globally by username.** `routes/auth.py:163` keys the rate-limit/lockout on `login:{username.lower()}` with no `schoolId` → across a multi-school deployment, one attacker hammering a common username (e.g. an owner's email reused across schools) locks that username out for **every** tenant. Separately, the primary lookup (`:184`) omits `schoolId` and relies on `get_db()` scoping to the env-default tenant, so a multi-school login that doesn't pass `school_id` resolves only against the default school — a second school's users can't log in without the client always supplying `school_id`.

- **P-M3 — Single-process assumptions across the runtime (multi-worker EB unsafe).** The SSE fan-out registry is an in-process dict (`services/sse.py:15`, comment acknowledges it); the AI kill-switch cache is per-process (AI-audit M8); `layaastat` fire-and-forget tasks aren't ref-held (AI-audit M9). EduFlow runs on Elastic Beanstalk where `WEB_CONCURRENCY > 1` is normal. Consequences: a notification/SSE event published on worker A never reaches a client connected to worker B; the kill-switch takes up to its TTL to propagate to other workers. The JWT-secret path already guards against this (`auth.py:37-43` refuses a per-process secret when `WEB_CONCURRENCY > 1`) — the same rigor isn't applied to the stateful in-process channels. Either document single-worker as a hard deploy constraint or move these channels to a shared broker.

- **P-M4 — `create_school` provisioning is non-atomic and partially rolls forward.** `operator.py:260-290` inserts `schools`, then `school_settings`, then `auth_users` as three separate un-transacted writes. A failure after the first leaves a half-provisioned school (registry row exists, no owner account) that also blocks re-provisioning via the `:254` duplicate check. No compensation/rollback.

- **P-M5 — Deactivated-school gate adds a DB read to every authenticated request and fails open.** `middleware/school_context.py:63-78` runs `db.schools.find_one({"school_id": ...})` on **every** non-skip request to check `status == "deactivated"`, uncached — a per-request round-trip on the hot path — and on any exception logs a warning and proceeds (fail-open), so a transient DB blip lets deactivated tenants keep transacting. Cache the status (short TTL) and decide the fail-open vs. fail-closed posture deliberately.

## 5. Low

- **P-L6 — Token debit (spend side) is non-atomic and can go negative.** `services/token_service.py:208-243` (`record_usage`) inserts the `token_usage` log and decrements `token_balances` as two separate try/except'd ops — either can fail independently (usage logged but not debited → free tokens, or debited but not logged → accounting drift). The `$inc {personal_topups.user_id: -tokens_used}` has no floor, so a balance can be driven negative. Spend-side mirror of the billing P-H2 atomicity gap; lower severity because plan-source quota is recomputed by aggregation, but the top-up sources are stateful.
- **P-L7 — Manual attendance entry 500s on a re-mark.** `attendance.py:106` inserts into `student_attendance` (unique index `(student_id, date)`) without catching `DuplicateKeyError`, so marking the same student/date twice returns a 500 instead of a friendly "already marked" / update. The bulk path (`mark_attendance` service) handles idempotency correctly; only the single manual-entry route doesn't. (Attendance docs also carry no `branch_id`, but attendance is uniformly school-wide-scoped today, so this is consistency, not a new leak — part of the P-H5 branch-scoping decision.)
- **P-L8 — `activities.list_houses` writes on a GET.** `activities.py:63-79` auto-seeds four house documents when none exist, inside a GET handler, with no unique index on house name — two concurrent first-time GETs can create duplicate house sets. Move seeding to an explicit idempotent step or add a unique index.
- **P-L9 — `assistant.py` LLM endpoint is uncapped and mis-reports failure.** `POST /api/assistant` (`assistant.py:208`) calls the Azure LLM with no rate limit and no token accounting (it never goes through `token_service.record_usage`), so any admin/teacher can drive uncapped Azure spend; and on `ok=False` it returns the `ai_unavailable` dict as `reply` with `success: True` (the AI-audit XM7 bug — tracked by AI epic R1.7). Add rate/token accounting; the false-success is fixed by R1.7.
- **P-L1 — Naive `datetime.now()` (no tz) persisted widely.** ~191 occurrences of `datetime.utcnow()`/`datetime.now()` across routes/services (`upload.py:145`, `fees.py`, `notification_service.py:45`, etc.) write tz-naive ISO strings, while the auth/token layer uses tz-aware `datetime.now(timezone.utc)`. Mixed naive/aware timestamps make cross-collection ordering and TTL math subtly wrong (mirrors AI-audit L2). Standardize on tz-aware UTC.
- **P-L2 — Idempotency middleware buffers the full response body for every keyed mutation** (`server.py:171-182`) — fine for JSON, but there is no size cap; a large non-streaming response is fully materialized in memory to store the replay copy.
- **P-L3 — `_check_ai` / readiness probes only the Azure *endpoint*, not the key** (`server.py:303-313`) — same blind spot as AI-audit C2; a key misconfig passes health checks. (Fix tracked in AI epic R9.1; noted here for the platform health surface `operator.py:389-399` which duplicates the same check.)
- **P-L4 — `federation.py` queries a `tenants` collection** (`:69`) that EduFlow does not populate — the endpoint returns `[]` in practice; harmless but dead. Confirm the federation consumer tolerates empty.
- **P-L5 — `seed_status` endpoint is unauthenticated** (`auth.py:469`) and returns row counts for `auth_users`/students/staff/classes — low-value information disclosure; gate it or confirm intentional.

## 6. Systemic conclusions

1. **Multi-tenancy is only half-wired.** `ScopedDatabase` handles the *request* path well, but the *non-request* paths — webhooks (P-C2), provisioning (P-C1/P-M4), and un-overridden collection methods (P-M1) — bypass or mis-derive the tenant context. Every write outside an authenticated request needs an explicit, tested `school_id`.
2. **Billing has no atomicity or correctness gate.** P-H1/P-H2 are real money-loss bugs with no test guarding the credit path. Billing deserves the same "reported outcome == true outcome" contract the AI-layer's R2 gives confirmed writes.
3. **RBAC is inconsistent between the fine-grained helpers and the coarse `require_role("owner","admin")` gates.** The audit-grade helpers (`require_access`, `require_owner_or_principal`) exist but the PII-heavy surfaces (exports, file-serve) use the coarse gate (P-H3/P-H4).
4. **Single-process assumptions contradict the EB multi-worker deployment** (P-M3) — the JWT layer already learned this lesson; the stateful channels haven't.

## 7. Coverage & residual

The findings above (P-C1…P-M6, P-L1…P-L6) are the result.

**Verified sound during the sweep:** `record_payment` idempotency (student|period|head key, service-layer enforced); `students.list_students` teacher-scoping (fail-closed to empty on no assignment) and `re.escape` on search; `search.py` `re.escape`; `decide_approval_request` delegating record-level authz to the service (`ApprovalAuthorizationError` → 403); refresh-token rotation CAS (`auth_tokens.consume_refresh_token`); the `(username_lower, schoolId)` unique index design.

**R15 line-level read completed (this pass):** `import_data.py`, `sms.py`, `payroll.py` + `fees.py` payroll/payment endpoints, `staff.py`, `attendance.py` (scoping helpers + intro), `issues.py` (scan: 34 scoped calls, incident-service-delegated — sound), `fees_service.record_payment` (idempotent + txn-enlisted — sound), `settings.py` (service-delegated with correct role gates — sound). New findings: P-H6, P-H7, P-M7, P-M8. **Verified sound:** `fees_service.record_payment` single-service parity + idempotency key; `attendance.py` class-teacher-only access gate (`_teacher_can_access_class`); `staff.create_staff`/`update_staff` delegating privileged-field gating to `staff_service`; `payroll.disburse` unique-index idempotency; `issues.py` and `settings.py` scoping/authz.

**Confirmatory pass COMPLETE (this read):** `notifications.py`, `queries.py`, `reports.py`, `activities.py`, `assistant.py`, `attendance.py` (manual + bulk-mark handlers), `student_service.py`, `discount_service.py`, `org_config_service.py`. **Honest bound:** this audit has read line-by-line the entire route surface and every domain service that carries a write path, and pattern-swept the rest for the established anti-patterns (auth gaps, tenant/branch scoping, regex injection, N+1, non-atomic money ops, broken session revocation, unguarded access). No audit can *prove* zero remaining bugs — the R11.1 eval corpus remains the behavioral backstop for the AI layer — but as of this pass there are **no known un-investigated paths at any severity**; everything found is captured in epics R12–R15.
